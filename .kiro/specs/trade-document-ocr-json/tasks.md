# Implementation Plan

- [x] 1. Foundation: 実行基盤と共通前提を整備する
- [x] 1.1 バックエンドの実行設定とAPI起動導線を整備する
  - FastAPI アプリ起動に必要な設定とルーター接続を定義する
  - API が起動し、ヘルス確認で稼働状態を確認できる
  - _Requirements: 1.1_
- [x] 1.2 処理結果の永続化モデルとマイグレーションを整備する
  - 処理ID、状態、正規化JSON、整合判定結果を保持できる保存構造を定義する
  - マイグレーション適用後に保存構造が作成されることを確認する
  - _Requirements: 1.4, 4.4, 5.3_
- [x] 1.3 フロントエンドの画面導線とAPIクライアント基盤を整備する
  - 書類投入画面と結果参照画面の遷移導線を定義する
  - API 呼び出し共通処理と状態管理の基盤を準備する
  - 画面から API 呼び出しを実行できる初期状態になる
  - _Requirements: 1.1, 1.3_

- [x] 2. Core: OCR・正規化・整合判定の機能を実装する
- [x] 2.1 (P) 書類受付APIと処理状態取得APIを実装する
  - 書類受付時に処理を開始し、一意の処理IDを返す
  - 処理状態取得で成功・要確認・失敗を返せるようにする
  - API の入力不正を適切に拒否できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 5.3_
  - _Boundary: DocumentIngestController_
- [x] 2.2 (P) OCR連携ゲートウェイを実装する
  - 書類画像を Ollama に渡し、抽出結果（テキスト/表情報）を受け取る
  - OCR 失敗時に呼び出し元が扱える失敗結果を返す
  - モック入力で抽出結果を取得できることを確認する
  - _Requirements: 2.1_
  - _Boundary: OllamaOcrGateway_
- [x] 2.3 (P) JSON正規化サービスを実装する
  - 抽出結果を標準キーへマッピングする
  - 標準化できない項目を未対応項目として明示する
  - 同義表現が同一キーへ正規化されることを確認する
  - _Requirements: 2.2, 2.3, 3.1, 3.2, 3.3_
  - _Boundary: JsonNormalizationService_
- [x] 2.4 (P) 数値整合判定サービスを実装する
  - 単価×数量と小計の一致可否を判定する
  - 小計・税・合計の一致可否を判定する
  - 不一致時に期待値と差異箇所を返す
  - _Requirements: 4.1, 4.2, 4.3_
  - _Boundary: ConsistencyChecker_
- [x] 2.5 処理ユースケースを実装しOCR→正規化→判定を統合する
  - OCR、正規化、整合判定、保存を単一フローで順序実行する
  - 出力JSONに整合判定結果を含める
  - 処理完了時に利用者が参照可能な結果状態になる
  - _Depends: 2.2, 2.3, 2.4_
  - _Requirements: 2.1, 2.3, 4.4, 5.1_

- [x] 3. Integration: UI/APIとエラーフローを結線する
- [x] 3.1 フロントで書類アップロードから結果表示までを結線する
  - 書類アップロード実行後に処理結果を取得し表示する
  - 正規化JSONと整合判定結果を同時に表示する
  - 不整合または抽出不足時に要確認状態を表示できる
  - _Depends: 1.3, 2.1, 2.5_
  - _Requirements: 5.1, 5.2, 5.3_
- [x] 3.2 エラー応答と状態遷移を統一する
  - 入力不正、抽出失敗、判定不能を区別したエラー応答を返す
  - UI 側で 400/422/500 に応じた表示分岐を行う
  - 利用者が再確認・再処理の必要性を判断できる状態になる
  - _Depends: 2.1, 2.5, 3.1_
  - _Requirements: 1.2, 2.2, 4.3, 5.2_

- [ ] 4. Validation: 要件を満たすテストで動作を検証する
- [x] 4.1 ドメイン/アプリケーションの単体テストを実装する
  - 正規化ルールと同義キー統一のテストを追加する
  - 明細判定と合計判定の一致/不一致テストを追加する
  - テスト実行で要件に対応する判定ロジックが検証される
  - _Requirements: 3.1, 3.2, 3.3, 4.1, 4.2, 4.3_
- [x] 4.2 API統合テストを実装する
  - 書類受付から処理ID返却までのフローを検証する
  - 結果取得で status と整合判定結果が返ることを検証する
  - 抽出不足ケースで未対応項目が返ることを検証する
  - _Requirements: 1.1, 1.4, 2.2, 4.4, 5.3_
- [ ] 4.3 E2E/UIテストを実装する
  - アップロードから結果確認までの主要導線を検証する
  - 不一致時に要確認状態が表示されることを検証する
  - 利用者視点で手動転記代替の最小フローが成立することを確認する
  - _Depends: 3.1, 3.2_
  - _Requirements: 1.1, 5.1, 5.2, 5.3_

- [x] 5. 画像主系 + テキスト補正副系を実装する
- [x] 5.1 PDF/Excel の入力前処理を実装する
  - PDF/Excel を画像主系へ統一する前処理を追加する
  - 補助テキストを同時に抽出できる
  - _Requirements: 6.1, 6.2_
- [x] 5.2 補助テキスト補正を実装する
  - OCR抽出結果に不足がある場合、補助テキストで補正する
  - 補正後も整合判定とステータス判定が動作する
  - _Requirements: 6.2, 6.3_
